## Purpose

Run a repeatable integrity sweep across eight dimensions and produce a **structured report** (a markdown list grouped by dimension) that other skills can act on. The skill does **not** fix issues itself — fixes belong to the content skill responsible for the area (page, faq, events, operational-facts, images, testimonials). Integrity checks are mostly build-time (Vitest) plus targeted greps on `site/dist/` after a clean build.

This skill is **not** for:

- Performance audits (see `skills/performance-check/SKILL.md` / **P9-10** when published).
- SEO strategy or content quality (see `skills/seo/SKILL.md` / **P9-13** when published).
- Third-party link monitoring that requires network access beyond HEAD-checks (lychee CI was removed as unreliable; do not reintroduce without explicit direction).
- Fixing broken things — report only. Pass findings to the relevant content skill.

## When to use

- A director or agent asks for a site audit.
- Before merging a large structural PR (new pages, renames, registry changes, navigation rewiring).
- After a wave of maintenance changes (e.g. several FAQ edits in a week) to make sure nothing has drifted.
- On demand when a suspected regression surfaces in GSC (see `skills/seo/SKILL.md` once published).

## Files touched

**Read-only.** The skill inspects files and outputs a report. No writes unless the user explicitly asks for them.

Primary reads:

- `site/dist/**/index.html` — built output, source of truth for internal-link resolution, hreflang emission, canonical emission, sitemap content, JSON-LD emission.
- `site/dist/sitemap-index.xml`, `site/dist/sitemap-0.xml` — generated by `@astrojs/sitemap`.
- `site/dist/_redirects` — Cloudflare redirect rules emitted from `site/public/_redirects`.
- `site/src/lib/routes/page-shell-registry.ts` (`PAGE_SHELL_PATHS`, `STOCKHOLM_SV_EN_PAIRS`, `BERLIN_DE_EN_STORY_PAIRS`, `BERLIN_EN_STORY_SEO_CANONICAL`).
- `site/src/lib/page-registry/page-body-registry.ts` (`PAGE_CUSTOM_BODY_PATHS`).
- `site/src/lib/chrome/navigation.ts` (`NAVIGATION_VARIANTS`).
- `site/src/data/page-shell-meta.json`.
- `site/public/_redirects` — pre-build source for the redirect dimension.
- `docs/url-matrix.csv` — the intended URL-keep/redirect inventory.
- `assets/images/photos.yaml` — curated image catalog for reference-integrity checks.

Primary tests (already present — rely on them; do not duplicate):

- `site/src/lib/routes/build-output-structure.test.ts` — every shell has a body, no orphan bodies.
- `site/src/lib/routes/page-shell-registry.test.ts` — registry covers every path in `page-shell-meta.json`.
- `site/src/lib/routes/url-matrix-parity.test.ts` — `url-matrix.csv` and the registry agree.
- `site/src/lib/routes/chrome-navigation-resolve.test.ts` — navigation resolves correctly across locales.

## Locale parity rules

- Stockholm pairs: every `sv` path in `STOCKHOLM_SV_EN_PAIRS` must have a matching `en` path built, with symmetric hreflang and mutual canonicals per `docs/Andetag SEO Manual.md` §hreflang.
- Berlin story pairs: every `de` path in `BERLIN_DE_EN_STORY_PAIRS` must have a matching `en` path built. Berlin English story shells canonicalise to Stockholm English per `BERLIN_EN_STORY_SEO_CANONICAL` — this is **expected asymmetry** documented in `docs/seo/decisions.md` `SEO-0016`; do not flag it.
- `x-default` required on Berlin English shells per registry test; confirm in the built HTML head.

## Workflow

### A. Produce a clean build first

```bash
cd site && npm test && npm run build
```

If tests or build fail, stop here and report that — integrity checks on a stale `dist/` are meaningless.

### B. Run the eight checks

Run each dimension below and capture findings. Each check should be fast (grep- or node-based); none require network calls by default.

#### B.1 Internal link resolution

Every `<a href="/...">` in built `dist/` must resolve to a live route in `dist/` or to a documented redirect in `dist/_redirects`.

```bash
cd site
# Collect every internal href in dist/
grep -rhoE 'href="/[^"#?]+' dist/ | sed 's/^href="//' | sort -u > /tmp/andetag-internal-hrefs.txt

# Collect every built route (as a directory with index.html)
find dist -name index.html -print | sed -e 's#^dist##' -e 's#/index\.html$#/#' | sort -u > /tmp/andetag-built-routes.txt

# Collect redirect sources
awk 'NF && $1 !~ /^#/ {print $1}' dist/_redirects | sort -u > /tmp/andetag-redirect-sources.txt

# Anything in hrefs that is not a built route and not a redirect source
comm -23 /tmp/andetag-internal-hrefs.txt <(cat /tmp/andetag-built-routes.txt /tmp/andetag-redirect-sources.txt | sort -u)
```

Report: each orphan href, paired with the first file that emits it.

#### B.2 External link health (optional HEAD-check)

Default: list external hrefs only, no network call. (The old `lychee` CI job was removed — see `CHANGELOG.md` **Removed**.)

```bash
grep -rhoE 'href="https?://[^"]+' dist/ | sed 's/^href="//' | sort -u
```

If the user explicitly asks for HEAD-checks, use `curl -I -L -o /dev/null -w "%{http_code} %{url_effective}\n" <url>` one URL at a time and cap at ~50 to avoid rate-limits. Flag: non-2xx status, `308`/`301` that chains more than once, and domain changes. Do **not** auto-flag TripAdvisor or social domains — they block bots aggressively and will false-positive.

#### B.3 Locale parity (pair symmetry)

```bash
# Stockholm sv ↔ en pairs
node -e '
const { STOCKHOLM_SV_EN_PAIRS } = require("./site/src/lib/routes/page-shell-registry.ts");
for (const [sv, en] of STOCKHOLM_SV_EN_PAIRS) {
  const svBuilt = require("fs").existsSync("site/dist" + sv + "index.html");
  const enBuilt = require("fs").existsSync("site/dist" + en + "index.html");
  if (!svBuilt || !enBuilt) console.log("ASYMMETRIC", sv, svBuilt, en, enBuilt);
}
'
```

(If the registry uses TypeScript-only syntax, run the same check as a Vitest add-on rather than raw Node. The existing `build-output-structure.test.ts` already approximates this — confirm both built paths exist.)

Report: any pair where one locale is missing.

#### B.4 Canonical and hreflang integrity

For each built page, confirm the head contains:

- `<link rel="canonical" href="…">` — self-canonical for normal pages; cross-canonical for Berlin English story shells (expected, not a bug).
- `<link rel="alternate" hreflang="sv" href="…">` and `<link rel="alternate" hreflang="en" href="…">` on Stockholm pairs; `hreflang="en"` + `hreflang="de"` + `hreflang="x-default"` on Berlin pairs.
- Symmetric references: page A's `hreflang="en"` points at B; B's `hreflang="sv"` points at A.

```bash
# Spot-check one Stockholm pair
grep -E 'rel="canonical"|rel="alternate"' site/dist/sv/stockholm/index.html
grep -E 'rel="canonical"|rel="alternate"' site/dist/en/stockholm/index.html
```

Rely on `page-shell-registry.test.ts` for structural coverage; this check catches drift between the registry and what actually ships in HTML.

#### B.5 Registry coherence

Confirm **`page-shell-registry`**, **`page-body-registry`**, `navigation.ts`, `_redirects`, and `url-matrix.csv` agree on the set of live routes.

- `page-shell-registry.ts` ↔ `page-shell-meta.json`: covered by `page-shell-registry.test.ts`.
- `page-shell-meta.json` paths ↔ `url-matrix.csv` kept-set: covered by `url-matrix-parity.test.ts`.
- Every `url-matrix.csv` row marked `redirect` must have a matching rule in `_redirects` (or its destination is the same origin's canonical). Spot-check:

  ```bash
  awk -F, '$2 == "redirect" {print $1}' docs/url-matrix.csv > /tmp/andetag-url-matrix-redirects.txt
  awk 'NF && $1 !~ /^#/ {print $1}' site/public/_redirects > /tmp/andetag-redirects-source.txt
  comm -23 <(sort -u /tmp/andetag-url-matrix-redirects.txt) <(sort -u /tmp/andetag-redirects-source.txt)
  ```

  Anything left is a documented redirect that has no rule. Report.

- Every route in navigation variants must exist in the shell registry. Spot-check by loading `NAVIGATION_VARIANTS` and confirming every href resolves to a `PAGE_SHELL_PATHS` entry.

#### B.6 Image reference integrity

Every image path referenced by `src` or `srcset` in `dist/` should resolve to a file under `dist/wp-content/uploads/` or elsewhere in `dist/`.

```bash
cd site
# Collect unique image paths from built HTML
grep -rhoE '(src|srcset)="[^"]*\.(jpg|webp|png|svg)"' dist/ \
  | sed -E 's/^(src|srcset)="//' -e 's/"$//' \
  | tr ',' '\n' | awk '{print $1}' | sort -u > /tmp/andetag-image-refs.txt

# Filter to internal paths starting with /
grep -E '^/' /tmp/andetag-image-refs.txt | while read -r p; do
  test -f "dist${p}" || echo "MISSING $p"
done
```

Cross-check the catalog: files named in `assets/images/photos.yaml` (`file:` field) should either appear under `site/public/wp-content/uploads/<year>/<month>/` with derivatives or be noted as unwired. A catalog entry with no consumers is not a bug; a `src` pointing at a nonexistent file is.

#### B.7 Sitemap canonicalness

`dist/sitemap-0.xml` should list only canonical, indexable, live routes.

```bash
grep -oE '<loc>[^<]+' site/dist/sitemap-0.xml | sed 's/^<loc>//' | sort -u > /tmp/andetag-sitemap.txt

# Every sitemap URL should be a built canonical route
while read -r url; do
  path="${url#https://www.andetag.museum}"
  test -f "site/dist${path}index.html" || echo "SITEMAP-NONEXISTENT $url"
done < /tmp/andetag-sitemap.txt
```

Also scan for `<meta name="robots" content="noindex">` in any HTML whose path appears in the sitemap — a noindexed URL in the sitemap is a regression.

#### B.8 Redirect chain length

Every rule in `_redirects` should be single-hop (source → destination, where destination is a live route).

```bash
awk 'NF && $1 !~ /^#/ {print $1, $2}' site/dist/_redirects \
  | while read -r src dst; do
      # Destination should be a built route or an external URL
      case "$dst" in
        https://*|http://*) echo "EXTERNAL $src -> $dst" ;;
        *) test -f "site/dist${dst}index.html" || echo "DEAD $src -> $dst" ;;
      esac
    done

# Any destination that also appears as a source = multi-hop
awk 'NF && $1 !~ /^#/ {print $2}' site/dist/_redirects | sort -u > /tmp/andetag-redirect-destinations.txt
comm -12 /tmp/andetag-redirect-sources.txt /tmp/andetag-redirect-destinations.txt
```

Report: dead destinations, any chains.

### C. Produce the report

Output format (copy into a PR comment, a maintenance-backlog row, or a chat reply):

```markdown
# Site integrity audit — YYYY-MM-DD

## Summary
- Build: pass / fail
- Tests: pass / fail (134/134 at time of writing)
- Dimensions with findings: N of 8

## Findings by dimension

### 1. Internal link resolution
- `<path>` — references `<href>` which does not resolve. First emitter: `dist/.../index.html`.

### 2. External link health
- (skipped unless HEAD-check requested) / `<url>` — HTTP `<code>`.

### 3. Locale parity
- `<pair>` — missing `<locale>`.

### 4. Canonical and hreflang
- `<page>` — missing `hreflang="xx"` / asymmetric reference.

### 5. Registry coherence
- `url-matrix.csv` row `<source>` marked redirect has no `_redirects` rule.

### 6. Image reference integrity
- `<path>` — referenced but not present on disk.

### 7. Sitemap canonicalness
- `<url>` — listed in sitemap but `noindex` in head / 404.

### 8. Redirect chain length
- `<source> -> <destination>` — destination itself is a redirect source (chain).
```

If a dimension has zero findings, write `- none.` — do not omit the heading. A clean audit is itself a useful artifact.

## Verification

The audit is itself a verification. There is nothing to re-verify post-audit unless fixes follow; fixes belong to the responsible content skill.

Automated coverage that already runs in CI (do not duplicate in this skill):

- `npm test` — 134 tests, includes registry, parity, and navigation coverage.
- `npm run build` — 65 pages generated; build fails on broken imports or typed route mismatches.

## When to escalate

Stop and ask before proceeding if:

- The build fails. Run the `page` or `rollback` skill on the failing change; this skill cannot audit a stale `dist/`.
- A finding suggests a shipped SEO regression on `www` (missing canonical, broken hreflang). Route to `skills/seo/SKILL.md`; if that skill cannot resolve it, escalate to Gustaf.
- The user asks for a full external-link HEAD-check sweep. The number of external links and the domains' bot-blocking make this fragile; confirm scope and live with false positives.
- Multi-hop redirects are found that were apparently intentional (e.g. two-stage locale migration). Do not collapse them without checking `docs/seo/url-architecture.md` and `docs/seo/decisions.md` (and the archived `docs/archive/migration-exceptions.md` for legacy migration deviations).
- The audit reveals a registry or sitemap anomaly that the existing tests did not catch. That is a test gap — document it in `docs/maintenance-backlog.md` before fixing the data.

## Examples

### Example 1: quick pre-merge audit before a large page rename

Action:

1. `cd site && npm test && npm run build`.
2. Run B.1, B.5, B.8 only (link resolution, registry coherence, redirect chains — the dimensions most affected by renames).
3. Output a short report. If B.5 surfaces a `url-matrix.csv` row with no redirect rule, point the user at `skills/page/SKILL.md` §rename workflow.

### Example 2: weekly audit

Action:

1. Full build + all eight dimensions.
2. Post the report as a maintenance-backlog row with `owner: skills/site-integrity`, status `done` (audit complete) — the actual findings become separate rows if they require fixes.

### Example 3: GSC flagged a `404` cluster

Action:

1. Skip B.2; focus B.1, B.5, B.7 (link resolution, registry coherence, sitemap canonicalness).
2. Cross-reference the GSC path list with the findings. If a GSC-flagged URL is neither in `dist/` nor in `_redirects`, report it with suggested next step: add a redirect (per `skills/page` rename workflow) or investigate whether the URL was ever valid.
